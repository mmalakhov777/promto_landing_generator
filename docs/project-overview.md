> Версия документа: 1.0

---

## 1. Что это такое

**Promto Landing Generator** — система автоматической генерации и управления типовыми SEO-оптимизированными лендингами для платформы [Promto](https://promto.ai) — ИИ-конструктора сайтов, сервисов, чат-ботов, трекеров и других цифровых продуктов.

Система разворачивается на поддомене **`types.promto.ai`** и представляет собой полнофункциональный веб-сервис с публичной частью (SEO-лендинги), админ-панелью и REST API.

---

## 2. Бизнес-цель и стратегия

### Проблема
Promto нуждается в привлечении органического поискового трафика по низко- и среднечастотным запросам, связанным с созданием сайтов для конкретных ниш: "создать сайт для педагога", "создать сайт автосервис", "сайт для банка" и т.д. Основной сайт `promto.ai` не может охватить всё многообразие таких запросов одной страницей.

### Решение
Создать сеть целевых лендингов — по одному на каждый поисковый запрос. Каждый лендинг максимально оптимизирован под конкретный ключевой запрос и ведёт пользователя к целевому действию — переходу на `app.promto.ai` для создания сайта.

### Воронка конверсии

```
Поисковый запрос (Яндекс/Google)
    ↓
SEO-лендинг на types.promto.ai
    ↓
Пользователь вводит промт / нажимает CTA
    ↓
Redirect на app.promto.ai?prompt=...&utm_source=types
    ↓
Регистрация → Создание сайта → Покупка подписки
```

### Ключевые метрики
- Органический трафик на `types.promto.ai`
- CTR из поиска (позиции в топ-10)
- Конверсия: лендинг → переход на `app.promto.ai`
- Конверсия: переход → регистрация → покупка (отслеживается на стороне `app.promto.ai`)

## 5. Технологический стек

### Архитектура

```
┌─────────────────────────────────────────────────┐
│              types.promto.ai                    │
│         (Next.js 14+ SSR — Docker)              │
│   Публичные лендинги + Админ-панель (/admin)    │
│              Port: 3000                         │
├─────────────────────────────────────────────────┤
│            API Backend (FastAPI)                 │
│          CRUD, Auth, Content API                │
│              Port: 8000                         │
├─────────────────────────────────────────────────┤
│       PostgreSQL 16 (managed DB)                │
│     Лендинги, контент, пользователи             │
└─────────────────────────────────────────────────┘
```

### Обоснование технологий

| Слой | Технология | Причина выбора |
|------|-----------|----------------|
| **Frontend/SSR** | Next.js 14+ (App Router, SSG + ISR) | Статическая генерация HTML при билде — золотой стандарт для SEO. ISR обновляет страницы без полной пересборки. Server Components минимизируют JS-бандл. Встроенная оптимизация изображений. |
| **Стили** | Tailwind CSS | Zero-runtime CSS, минимальный бандл, дизайн-система по брендбуку Promto. |
| **Backend API** | FastAPI (Python) | Совпадает со стеком `promto-blog`, async-нативный, response <50ms, автогенерация OpenAPI-документации. |
| **ORM** | SQLAlchemy 2.0 + Alembic | Async через asyncpg, миграции, совместимость с `promto-blog`. |
| **База данных** | PostgreSQL 16 | JSONB для гибкого хранения контентных блоков, полнотекстовый поиск для админки. |
| **Локализация** | next-intl | Лёгкая, поддерживает App Router, совместима с SSG. |
| **SEO-инструменты** | next-sitemap, JSON-LD (Schema.org) | Автогенерация sitemap.xml, структурированные данные для поисковых систем. |
| **Антибот** | Yandex SmartCaptcha | Невидимая капча, не ухудшает UX, защищает формы от ботов. |
| **Аналитика** | Яндекс.Метрика | Требование заказчика. Вебвизор + кастомные цели. |
| **Деплой** | Docker + docker-compose | Деплой через Promto (type: backend для SSR Next.js + бэкенд-сервис + managed PostgreSQL). |

### Почему именно этот стек (ключевые решения)

| Решение | Альтернатива | Обоснование |
|---------|-------------|-------------|
| SSG + ISR (не чистый SSR) | Server-side rendering | SSG отдаёт готовый HTML мгновенно — лучший PageSpeed. ISR обновляет без ребилда всего сайта. |
| JSONB для контента (не отдельные таблицы) | Нормализованные таблицы на каждый тип блока | Гибкость: новые поля без миграций. Производительность достаточна при <10K лендингов. |
| Монорепо (frontend + backend) | Отдельные репозитории | Проще деплой через docker-compose, атомарные изменения контракта. |
| Админка внутри Next.js (/admin) | Отдельное React SPA | Меньше сервисов, переиспользование компонентов, проще деплой. Закрыта от индексации в robots.txt. |
| SEO встроен в компоненты (не отдельная фаза) | Сначала страницы, потом SEO | Мета-теги, Schema.org, структура заголовков — неотъемлемая часть компонентов. Делать "потом" = переделывать. |

## 6. Структура лендинга

Каждый лендинг построен по принципу **AIDA** (Attention → Interest → Desire → Action) и содержит 11 секций:

| # | Секция | Назначение | Обязательная |
|---|--------|-----------|:------------:|
| 1 | **Hero** | H1 с ключевым запросом + подзаголовок + поле ввода промта (имитация чата с ИИ) + CTA-кнопка → app.promto.ai | Да |
| 2 | **Social Proof Bar** | Счётчики: "X сайтов создано", "Y пользователей", рейтинг | Нет |
| 3 | **Преимущества** | 4–6 ключевых выгод с иконками, адаптированных под нишу | Нет |
| 4 | **Как это работает** | 3–4 шага работы с конструктором | Нет |
| 5 | **Примеры работ** | Галерея/карусель готовых сайтов по нише | Нет |
| 6 | **CTA-блок (промежуточный)** | Повторное поле ввода промта + кнопка | Нет |
| 7 | **Видео** | Демо-видео работы конструктора | Нет |
| 8 | **Тарифы** | Карточки тарифов с CTA-кнопками → app.promto.ai | Нет |
| 9 | **Отзывы** | Карусель отзывов с Schema.org разметкой Review | Нет |
| 10 | **FAQ** | Аккордеон вопросов-ответов с FAQPage Schema.org | Нет |
| 11 | **Финальный CTA** | Большой блок с полем ввода промта + сильный призыв к действию | Да |

**Хлебные крошки** отображаются над H1 на всех страницах (Главная → Категория → Лендинг).

### Hero-секция — ключевой конверсионный элемент

Hero содержит поле ввода, имитирующее чат с ИИ. Пользователь вводит описание желаемого сайта (например: "Опишите, какой сайт для школы вы хотите создать..."), нажимает кнопку "Создать", и перенаправляется на `app.promto.ai` с промтом в query-параметре и UTM-метками:

```
app.promto.ai?prompt={текст}&utm_source=types&utm_medium=landing&utm_campaign={category_slug}&utm_content={landing_slug}
```

Это сокращает путь пользователя: он уже ввёл запрос и попадает сразу в процесс создания.

## 7. SEO-стратегия

SEO — ключевой приоритет всего проекта. Каждое техническое решение принималось с учётом влияния на поисковую оптимизацию.

### 7.1 URL-структура

```
types.promto.ai/
├── /ru/                                    # Главная (RU)
├── /en/                                    # Главная (EN)
├── /ru/site-generator/                     # Категорийная страница (листинг)
├── /ru/site-generator/page2/               # Пагинация категории
├── /ru/site-generator/pedagoga/            # Лендинг конкретной ниши
├── /en/site-generator/pedagoga/            # EN-версия того же лендинга
├── /robots.txt                             # Правила индексации
├── /sitemap.xml                            # Карта сайта
└── /admin/                                 # Админ-панель (закрыта в robots.txt)
```

Правила URL:
- Только латиница, только тире (без `_`, `%`, `?`, спецсимволов)
- Обязательный trailing slash (301-редирект с `/path` на `/path/`)
- Нормализация множественных слешей (`///` → `/`)
- ЧПУ без лишних вложений (`/site-generator/pedagoga/`, а не `/category/site-generator/page/pedagoga/`)

### 7.2 Мета-теги и OpenGraph

- `<title>`: `{H1} — Промто` (RU) / `{H1} — Promto` (EN), < 60 символов
- `<meta name="description">`: уникальное описание < 160 символов с ключевым запросом
- OpenGraph: `og:title`, `og:description`, `og:image`, `og:url`, `og:type=website`, `og:locale`
- Canonical URL на каждой странице
- Hreflang: парные ссылки `ru` ↔ `en`, `x-default` → `ru`
- Пагинация: `title \- страница N` для страниц > 1

### 7.3 Schema.org (структурированные данные)

JSON-LD разметка в `<head>` каждого лендинга:

| Тип | Где применяется |
|-----|----------------|
| `BreadcrumbList` | Все страницы кроме главной |
| `FAQPage` | Лендинги с включённой секцией FAQ |
| `Product` + `Offer` | Лендинги с секцией тарифов |
| `Review` + `AggregateRating` | Лендинги с секцией отзывов |
| `Organization` | Глобально (footer/layout) |

### 7.4 Технические SEO-требования

Проект реализует 42 SEO-требования из документа `Общие seo требования при разработке сайта promto.ai`. Ключевые:

- **robots.txt**: закрыть `/admin/`, тестовые версии, результаты поиска; открыть всё остальное
- **sitemap.xml**: авто-генерация без `lastmod`/`changefreq`/`priority`, исключение закрытых в robots.txt
- **404-страница**: с навигацией сайта, `<h1>Данная страница не существует!</h1>`, ссылка на главную
- **301-редиректы**: trailing slash, множественные слеши, http→https, www→non-www, slug-изменения
- **Заголовки**: единственный `<h1>` на странице, `<h2>`–`<h4>` только в контенте, без `<h*>` в header/footer
- **Изображения**: WebP, lazy loading, alt-теги на всех `<img>`
- **Внешние ссылки**: `rel="nofollow noopener"`
- **Производительность**: PageSpeed > 90 (mobile + desktop), время ответа < 200ms, Core Web Vitals в зелёной зоне
- **Валидация**: W3C HTML/CSS, валидатор микроразметки Яндекс

### 7.5 Производительность

| Метрика | Целевое значение |
|---------|-----------------|
| PageSpeed Insights (mobile) | > 90 |
| PageSpeed Insights (desktop) | > 90 |
| Время ответа сервера | < 200ms (оптимум < 50ms) |
| LCP | < 2.5s |
| FID / INP | < 200ms |
| CLS | < 0.1 |

Достигается за счёт:
- SSG (готовый HTML без серверного рендера при запросе)
- React Server Components (минимум JS на клиенте)
- Tailwind CSS (zero-runtime, tree-shaking)
- `next/image` (WebP, srcset, lazy loading)
- `next/font` (preload, display=swap, subset=latin+cyrillic)
- Видео: poster + iframe загрузка по клику
- ISR кеширование (revalidate: 3600)

### Auth-паттерн: JWT в httpOnly cookie

Проблема: httpOnly cookie нельзя установить из клиентского JavaScript.

Решение: Next.js API Routes как auth-прокси:
1. Клиент → `POST /api/auth/login` (Next.js API Route)
2. API Route → `POST /api/v1/auth/login` (FastAPI, server-to-server)
3. API Route получает JWT, устанавливает `Set-Cookie: httpOnly` и возвращает клиенту
4. Браузер автоматически прикладывает cookie к последующим запросам

## 11. Генерация контента через AI

### Процесс
1. Структура лендингов импортируется из Excel (slug, keyword, search_volume)
2. Скрипт `generate_content.py` для каждого лендинга формирует промт с:
   - Ключевым запросом лендинга
   - Информацией о платформе Promto
   - Целевой аудиторией по нише
   - Инструкциями по формату каждого поля (JSON-структуры)
3. AI генерирует уникальный контент: SEO-теги, тексты секций, FAQ, отзывы и т.д.
4. Контент сохраняется в LandingContent
5. EN-контент генерируется на основе RU через перевод + адаптацию
6. Контент-менеджер проверяет через админку и публикует

### Генерируемые поля
- SEO: meta_title (<60 символов), meta_description (<160 символов), H1, OG-теги
- Hero: заголовок, подзаголовок, текст CTA, placeholder промта по нише
- Преимущества: 4–6 карточек, адаптированных под нишу
- Как это работает: 3–4 шага
- FAQ: 5–6 вопросов-ответов
- Отзывы: 4–5 шаблонных отзывов
- CTA-тексты с призывом к действию по нише

### НЕ генерируемые поля (из SiteSettings)
- Social Proof (общие цифры платформы)
- Тарифы (единые для всех лендингов)
- Видео (общее демо)
- Примеры работ (общие скриншоты)

### Защита от ошибок
- `--resume`: продолжение генерации с места остановки
- `--dry-run`: оценка стоимости без API-вызовов
- Progress tracking: какие лендинги уже обработаны
- Rate limiting: пауза 1–2 секунды между запросами
- Retry с exponential backoff (3 попытки)
- Модель: Claude Sonnet (баланс цена/качество, ~$5–10 за 124 генерации)

## 14. Деплой и инфраструктура

### Платформа деплоя
Промто (через `promto.yaml`):
- **web** (Next.js SSR): type backend, framework docker, port 3000
- **api** (FastAPI): type backend, framework docker, port 8000
- **db**: type database, framework postgresql, preset_id 357 (managed)

### Docker-структура
- `frontend/Dockerfile` — Next.js standalone build
- `backend/Dockerfile` — FastAPI production
- `docker-compose.yml` — orchestration (frontend + backend + postgres)
- `docker-compose.dev.yml` — override с volume mounts для hot-reload

### ISR-ревалидация
- Автоматическая: `revalidate: 3600` (каждый час)
- On-demand: при сохранении лендинга в админке → backend вызывает `POST /api/v1/revalidate` → Next.js `revalidateTag()`

### Решение проблемы SSG + Docker build
При `docker build` backend недоступен, и `generateStaticParams()` не может получить список slug-ов. Решение:
1. **Код**: try/catch в `generateStaticParams()`, при ошибке → `[]` + `dynamicParams = true` (ISR при первом запросе)
2. **CI**: предварительный экспорт `static-params.json` из работающего API перед Docker build

## 15. Роутинг (архитектурное решение)

### Проблема
В Next.js App Router есть конфликт между пагинацией (`[[...page]]` catch-all) и slug лендинга (`[slug]` dynamic segment) — оба конкурируют за один URL.

### Решение
Единый catch-all маршрут `[...rest]/page.tsx` под `[category]/`:
- Если `rest[0]` соответствует паттерну `page\d+` → рендерить листинг с пагинацией
- Иначе → рендерить лендинг с `slug = rest[0]`

Дополнительно: запрет slug-ов вида `pageN` при создании лендингов.

## 16. Безопасность

- JWT-авторизация с httpOnly cookie (не localStorage)
- Серверная валидация SmartCaptcha-токенов
- CORS ограничен разрешёнными доменами
- Валидация slug-ов (зарезервированные значения: `admin`, `api`, `ru`, `en`, `_next`, `robots.txt`, `sitemap.xml`, `favicon.ico`)
- Pydantic-модели для валидации JSONB-структур на бэкенде
- XSS-защита через React (автоматическое экранирование)
- SQL-injection защита через SQLAlchemy ORM
- Внешние ссылки: `rel="nofollow noopener"`
- HTTPS с 301 редиректом с HTTP
